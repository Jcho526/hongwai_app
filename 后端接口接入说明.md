## 2. “每账号对应自己的数据库数据”是怎么实现的

并不是每个账号一个物理数据库（那样维护成本很高），而是：

- 所有人共用一个数据库
- 用 `user_id` 做数据隔离

具体：

1. 登录成功后，JWT 里带上 `uid`
2. 人员查询时：`WHERE user_id = 当前uid`
3. 人员修改/删除时：`WHERE id = ? AND user_id = 当前uid`
4. 系统设置表 `system_settings` 对 `user_id` 做唯一约束（一人一份）

这样就保证：A 账号看不到 B 账号的数据。

## 3. 数据库初始化（MySQL）

SQL 文件位置：`server/sql/init.sql`

包含三张核心表：

1. `users`：账号表（密码 hash 存储）
2. `persons`：人员档案（含 `user_id` 外键）
3. `system_settings`：系统设置（`user_id` 唯一）

### 3.1 执行方式（任选其一）

#### 方式A：在数据库工具中执行

## 5. 账号注册与登录教程

### 5.1 首次注册账号

接口：`POST /api/auth/register`

请求体：

```json
{
  "username": "admin",
  "password": "123456",
  "nickname": "管理员"
}
```

### 5.2 登录

接口：`POST /api/auth/login`

请求体：

```json
{
  "username": "admin",
  "password": "123456"
}
```

返回里会有 token，前端已自动保存并用于后续请求。

## 6. 人员档案 CRUD 接口说明

都需要登录态（即请求头带 Bearer token）。

1. 列表查询
   - `GET /api/persons?page=1&pageSize=20&keyword=&searchType=name`

2. 新增
   - `POST /api/persons`

3. 更新
   - `PUT /api/persons/:id`

4. 删除
   - `DELETE /api/persons/:id`

字段映射：

- 前端 `relativePhone` ↔ 表字段 `relative_phone`
- 前端 `idCard` ↔ 表字段 `id_card`

---

## 7. 系统设置接口说明

1. 获取设置：`GET /api/system/settings`
2. 保存设置：`POST /api/system/settings`
3. 版本检查：`GET /api/system/version/check`

设置数据同样按 `user_id` 隔离。

## 8. 前端联调配置

文件：`services/http.js`

当前建议：

```js
const BASE_URL = "http://127.0.0.1:3000";
const USE_MOCK = false;
```

### 8.1 真机调试注意

真机时 `127.0.0.1` 指手机自己，不是电脑。

请改成电脑局域网 IP，例如：

`http://192.168.1.100:3000`

## 9. 标准返回格式（后端统一）

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

- `code=0` 成功
- 非 0 为业务错误

---

## 10. 一键排错清单

联调失败时按顺序查：

1. 后端是否启动成功（health 接口）
2. `.env` 数据库配置是否正确
3. 数据库是否已执行 `init.sql`
4. 前端 `BASE_URL` 是否正确
5. 登录后本地是否保存了 token
6. 请求头里是否有 `Authorization: Bearer ...`
7. 接口报 401 是否 token 过期/无效
8. 手机与电脑是否同一局域网
